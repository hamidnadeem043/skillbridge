# SkillBridge

A full-stack freelance marketplace (Fiverr-style) built with the MERN stack. Buyers and sellers can browse gigs, chat in real time, complete secure payments, and leave reviews — all in one platform.

**Live Demo:** https://skillbridge-gamma-eight.vercel.app  
**Backend API:** https://skillbridge-production-cfdd.up.railway.app  
**GitHub:** https://github.com/hamidnadeem043/skillbridge

---

## Features

- **Authentication** — JWT-based auth with httpOnly cookies, bcrypt password hashing
- **Gigs** — Create, browse, search, filter, and delete service listings
- **Image Uploads** — Cloudinary integration for gig cover images
- **Payments** — Full Stripe checkout flow with payment intents
- **Orders** — Separate buying and selling views for every user
- **Real-Time Chat** — Socket.io powered instant messaging between buyers and sellers
- **Reviews & Ratings** — Verified buyers can rate and review completed gigs
- **Live Notifications** — Unread message badge updates in real time

## Core Data Models

- **User** — username, email, password (hashed), isSeller, country, profilePic
- **Gig** — title, description, price, category, deliveryTime, features, rating stats
- **Order** — gigId, buyerId, sellerId, price, payment_intent, isCompleted
- **Conversation** — sellerId, buyerId, lastMessage, read status per side
- **Message** — conversationId, userId, desc
- **Review** — gigId, userId, star, desc

## Key Engineering Decisions

**Why both buyerId and sellerId are stored on Order**  
Avoids an extra database lookup through the Gig collection on every order read — a deliberate denormalization for read performance.

**Why every user can both buy and sell**  
Initial design only showed one order view per role (`isSeller` decided buyer vs seller view). Fixed to always return both `sellingOrders` and `buyingOrders`, since `isSeller` is just a profile flag, not a hard restriction. Mirrors real marketplaces where anyone can transact in both directions.

**Conversation ID strategy**  
`conversationId` is generated by concatenating sellerId + buyerId consistently (always sellerId first), so the same two users always land on the same conversation thread regardless of who initiates it — avoids duplicate conversation documents.

**Cross-domain auth in production**  
Frontend (Vercel) and backend (Railway) are on different domains. JWT cookies needed `sameSite: 'none'` and `secure: true` to be accepted cross-site over HTTPS, plus explicit CORS `origin` allow-listing on both Express and Socket.io.

**Why Railway over Vercel for the backend**  
Socket.io requires a persistent, long-lived server connection. Vercel's serverless functions spin up per-request and don't support persistent WebSocket connections, so the backend needed a traditional always-on server (Railway).

## Known Issues / Trade-offs

- Local MongoDB used in development due to an Atlas SRV DNS resolution issue on the dev machine; Atlas works correctly in the deployed environment.
- Stripe integration is in test mode (Pakistan is not a supported Stripe payout country).
- No refresh-token rotation yet — JWT is long-lived (7 days) for simplicity.
